The aim ofthe Clean Architecture(CA) concept is to prevent duplicating code in multiple projects, which promotes good performance by reducing solution size, makes maintenance and development easier. This is because CA centralises the main business logic in the DOMAIN project which is then implemented in the appropriate projects for this reason every solution must contain the Domain Project. The other projects include User Interface (App Project), Infrastructure (MinAPI Project) using OpenAPI / Swagger and Unit Testing (UnitTesting Project). On creating a new solution the required projects must be added and the appropriate packages and references added. Once this is done the solution should be run before any development has taken place, to ensure that the projects are wired correctly.
The following assumptions have been made, These are :-

  1.  Microsoft Visual Studio 2022 - Has been installed.
  2.  Microssoft SQL Server - Has been installed.
  3.  Microssoft SQL Server Management Studio - Has been installed.
  4.  You are familiar with the installed software.

The purpose of the Templates Solution is to provide a starting point for future projects, depending on the agreed standards. The Templates Solution contains several different templates, these are :-

  1.  Application - The Application template incorporates the MVC framework with Serilog for logging, MS SQL Server database, not forgetting global exception handling, etc. For further information see the "App" tab.
  2.  Minimal API - . This is one of the other startup Project and deals with Database access using OpenAPI / Swagger. For further information see the "Min API" tab.
  3.  Unit Testing - The Unit Testing Template uses Fluent Assertions framework to test the application Project. For further information see the "Unit Testing" tab.
  4.  Domain - Contains all the Business Logic, Models, etc. For further information see the "Domain" tab


 FrameWork - Models-Views-Controllers (MVC) Project
It is fair to say that the MVC framework can be used to do all the tasks required in a project having said that the purpose of the "App Project" is to act as a start-up project and hub so that everything goes through the app but elements such as database access, logging & testing are done by specialist projects as and when required which provides consistency throughout solutions and projects within the solution. The App project contains the Models which can be created from either an existing database or from code.

  •  Models  :- Have been moved to the Domain Project.
  •  Views  :-  These are all the Front end HTML5 files (GUI).
  •  Controllers  :- Again the controllers for each screen.


 Endpoints :- All endpoints for each table depending on requirements, the class can be organised by using regions
 Framework - Web API Project
The purpose of the Minimal Application Programmable Interface project is to provide a mechanism to access the database securely. Without giving access to the database or divulging its structure unnecessarily, meaning you are in control of the Data source. Especially, as the Project uses Swagger then the API calls can be tested and documented in development mode, Each API can be as open or restricted as necessary to perform the required tasks for example minimal API's can be produced for the same solution with an Internal version for support and an External version a public user, thus ultimately protecting the database from outside sources with malicious intent and protecting corporate.


Serilog has been used to replace the built in logger because it is more Flexible and uses better logging descriptions.

  •  Overview - The App project is configured to use Serilog the appropriate Sink package is installed and the configuration is done then the logging will be done. Note all HTTP requests are logged automatically without the need to include the log entries in each endpoint or controller.
  •  Sinks - The main Sink (Serilog.Sinks.{Destination}) packages are below but the examples below are not exhaustive if your requirement is not accomodated below check the Serilog Documentation. some of the common Destinations are below :-
    1.  Console - Command Console Window.
    2.  Debug - Debug Window.
    3.  EventLog - Directly To the Windows Event Viewer.
    4.  Email - a nominated Email address.
    5.  File - A Text file in either Text or JavaScript Object Notation (JSON) format.
    6.  MSSQLServer - MS SQL Server Database.
  •  Configuration - All the Destination Packages can be configured independently in appsettings.json, or its environment variable equivalent. This adds to the flexibility of Serilog, My personal approach is to use the environment appsettings file for configuration that is necessary for the solution while the appsettings.json file is used solely for the user options configuration.
  •  Exception Handlers - In the exception Handler class all the relevant exception Handlers should be defined as appropriate and this is a rolling class ensuring consistency throughout projects.
The Logging project means that everything can be pre-defined and the development team only has to deal with making the log entries in the solution.


 The Domain project contains all classes in the following categories. The methods should be unit tested where necessary ensuring the solution is robust.

  • Functions  :- This relates to required functionality of a web site, traditionally Javascript was used but a language such as C# is pr-compiled and hence is compressed and executes faster, not forgetting more secure.
  • Niddleware  :- This is the location of all the custom Middleware.
  • Models  :- These are models that are used by the solution and created during development, generated Models should not be moved to this location.
  • Services  :- This is the location of all the custom Middleware.
  • Validators  :- These are classes that applies the rules for the endpoints used by the solution.
  • wwwroot  :- This is the location of all standard software.


  FrameWork - Xunit & Fluent Assertions.
  Tests :- Each class (This includes, Functionality, Validators, Exception Handlers, etc.) should have a test Class. These are usually located in the Domain class and should have a test Method. The proposed naming convention is as follows :-

  1. Test Class name = Class name + 'Tests'.cs so if the class to be tested is Sample.cs the Test Class will be called SampleTests.cs
  2. Method name = Method Name in the class with the same parameters with the return value. Method Name + '_' + Return type so a method with the return type 'string' and called SampleMethod(int input) is called SampleMethod_String.
 This Project approaches testing by having 2 Example Tests (1 Fact & 1 Theory) and is wired up to the app project it uses to define the criteria meaning that this is a good starting point for Unit Testing in any future project also don't forget to update the GlobalUsings.cs to include new projects and/or folders as required. To run you just have to "Run All Tests" from the "Test" menu and view the result in the Test Explorer. Fluent assertions provides meaningful results when run.

  All software must meet two criteria, these are :-

  1. Functionality Requirements - Unit Tests should Test every case scenario to ensure the functionailty is fully met, and some incorrectly called instances to ensure the software is robust.
  2. Robust - software must be able to deal with all other scenarios for example by throwing appropiate exceptions and logging them the reason for this is to deal with any security issues / attempted security breaches.


 How do you create a Template based on a Project?

  1.  Clone Project -  Go to the Github repository and clone the project so that it exists locally. See Link clone
  2.  Amend Project -  Make any changes to the project so that it meets your needs.
  3.  VS2022 -  Navigate to Project > Export Template...
  4.  Window 1 -  Select Template type Project and Project name from dropdown > Next button.
  5.  Window 2 -  Enter the Template Name, Description & Location > Finish button.
  6.  Installed -  Template will now be visible when creating a new project. Alternatively, See link create a template
  7.  Item Template -  Select the item option in window 1, and follow the wizard appropriately.
  8.  Update - After making changes re save the template with the same name and that template will be over-written
