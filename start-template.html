Testing has been a major part of software development from day 1 however, testing methodologies has changed. What are the advantages of Unit Testing?

  1. Balanced Approached - Tests should be wrote ahead of the source code from the specification which outlines the parameters of the associated method. Only dynamic(Can be changed) internal(Not external Libraries or Frameworks) methods should be tested.
  2. Encourage Good Coding Practices - Unit Tests should only be done on methods which are Testable (one unit of work), meaning a method which in the past did several units of work then that method is not testable, however by breaking the method down into manageable components each component can be tested. As a result the quality of code is improved.
  3. Expose Security Vulnerabilities - The most common secutity vulnerability is the "one up, one down vulnerability" which in short enables code to be exploited when for example a loop does one too many or few iterations, if these can be exposed then the software is inherently more secure.
  4. Expose newly introduced problems - If a change causes a test to fail this draws attention to the change and means that both the test and change should be reviewwed as the test may be too strict or the change does in fact take the method outside of the specified parameters of the method meaning that potential issues are exposed. Whereby in the past this would have been missed.
This Project approaches testing by having 2 Example Tests (1 Fact & 1 Theory) and is wired up to the app project it uses Fluent Assertions to define the criteria meaning that this is a good starting point for Unit Testing in any future project also don't forget to update the GlobalUsings.cs to include new projects and/or folders as required. To run you just have to "Run All Tests" from the "Test" menu and view the result in the Test Explorer. Fluent assertions provides meaningful results when run.

The Naming convention of unit testing is a s below

  1. Unit Testing Project - This is simply called Unit Testing.
  2. Project being tested - This is a new folder called "UT.{Projectname}".
  3. Class being tested - A Class be created within the project folder called "{ClassName}Tests".
  4. Method being Tested - a method should be created with the name "{methodname}_{ReturnType}" this ensures all overloads can be tested independently.
The above naming convention allows a structure to the tests and indicates exactly what is being tested in the results making life easier for developers. If you are new to Unit Testing I would suggest you view a youtube video in conjunction with the examples in the start-template.UnitTesting project.
